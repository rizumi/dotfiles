## 1. 一貫性チェック

### 目的
同じ概念を表す文言が統一されているか確認

### チェック内容

#### 1.1 同一キーの訳語統一
同じ英語キー/原文が異なる訳語になっていないか確認

**チェック方法:**
```bash
# 特定のキーを横断的に検索
grep -r "\"Cancel\"" **/*.xcstrings
grep -r "\"cancel\"" **/*.json
grep -r ">Cancel<" **/strings.xml
```

**問題例:**
```
File1: "Cancel" → "キャンセル"
File2: "Cancel" → "取消"
File3: "Cancel" → "中止"
```

**推奨:**
プロジェクト全体で統一（例：すべて「キャンセル」）

#### 1.2 類似機能での表現統一
類似の機能で異なる表現が使われていないか確認

**問題例:**
```
選択ボタン: "選択", "選ぶ", "選んでください"
削除ボタン: "削除", "削除する", "消去"
```

**推奨:**
同じアクションには同じ表現（例：すべて「選択」「削除」）

#### 1.3 アクション表現の統一
動詞形と名詞形が混在していないか確認

**問題例:**
```
ボタン1: "保存" (名詞形)
ボタン2: "削除する" (動詞形)
ボタン3: "キャンセル" (名詞形)
```

**推奨:**
ボタンラベルは名詞形で統一

### チェック手順

1. プロジェクト内のすべての翻訳ファイルを特定
2. 頻出キー（Cancel, Delete, Save, Select等）をリストアップ
3. 各キーの訳語を横断的に検索
4. 異なる訳語が使われている箇所を特定
5. 最も適切な訳語を決定し、統一を推奨

## 2. 用語統一チェック

### 目的
技術用語やブランド名が統一された表記になっているか確認

### チェック内容

#### 2.1 ブランド名の表記
固有名詞が原文のまま維持されているか確認

**確認項目:**
- Apple製品名: Face ID, Touch ID, App Store, iCloud
- 一般的なブランド: GitHub, Google, Facebook
- プラットフォーム名: iOS, Android, Windows

**問題例:**
```
正: "Face ID"
誤: "フェイスID", "FaceID", "face id"
```

#### 2.2 技術用語の統一
技術用語が一貫した表記になっているか確認

**日本語の場合:**
```
推奨: "ログイン" (一貫性)
避ける: "ロクイン", "log in" の混在

推奨: "スライドショー"
避ける: "スライド ショー", "slideshow" の混在
```

**韓国語の場合:**
```
推奨: "슬라이드쇼"
避ける: "슬라이드 쇼", "slideshow" の混在
```

#### 2.3 カタカナ表記の統一
長音符の使用が統一されているか確認（日本語）

**確認項目:**
```
推奨: "データ", "サーバー", "ユーザー"
避ける: "データー", "サーバ", "ユーザ" との混在
```

#### 2.4 外来語の表記選択
カタカナと漢字の使い分けが統一されているか確認

**例:**
```
選択肢1: "キャンセル" (カタカナ)
選択肢2: "取消" (漢字)
→ プロジェクト全体でどちらかに統一
```

### チェック手順

1. プロジェクトの`TRANSLATION_GUIDELINES.md`や`TERMINOLOGY.md`を確認
2. 存在しない場合は、頻出用語をリストアップ
3. 各用語の表記を横断的に検索
4. 表記揺れを特定
5. 統一基準を提案

## 3. 文体・トーンチェック

### 目的
文体やトーンが統一され、適切か確認

### 日本語の文体チェック

#### カテゴリ別の文体基準

| カテゴリ | 文末表現 | 例 |
|---------|---------|-----|
| ボタンラベル | 名詞形・動詞形 | 完了、キャンセル、選択 |
| 画面タイトル | 名詞形 | アプリ情報、設定 |
| 指示メッセージ | 丁寧な命令形 | ～してください |
| 状態メッセージ | です・ます調 | ～です、～しました |
| エラーメッセージ | です・ます調 | ～できません |
| 確認ダイアログ | 疑問形 | ～しますか？ |

#### チェックポイント

**敬語レベルの統一:**
```
問題例:
メッセージ1: "選択してください" (丁寧語)
メッセージ2: "選択しろ" (命令形)
メッセージ3: "選択して" (普通形)

推奨: すべて丁寧語で統一
```

**カテゴリごとの適切性:**
```
ボタン: "完了" ✓
ボタン: "完了してください" ✗ (冗長)

エラー: "削除できません" ✓
エラー: "削除失敗" ✗ (不親切)
```

### 韓国語の文体チェック

#### 존댓말（丁寧語）の統一
```
推奨: "선택해 주세요" (丁寧)
避ける: "선택해" (非丁寧) との混在
```

#### 격식체（格式体）の使い分け
```
フォーマル: "선택하십시오"
カジュアル: "선택하세요"
→ プロジェクトのトーンに合わせて統一
```

### 英語の文体チェック

#### フォーマル vs カジュアル
```
フォーマル: "Please select an option"
カジュアル: "Pick one"
→ プロジェクトのトーンに合わせて統一
```

#### 命令形 vs 依頼形
```
命令形: "Click here"
依頼形: "Please click here"
→ プロジェクトのトーンに合わせて統一
```

### チェック手順

1. プロジェクトのトーン（フォーマル/カジュアル）を確認
2. カテゴリ別に文言をサンプリング
3. 文末表現をチェック
4. 不統一な箇所を特定
5. 統一基準を提案

## 4. プレースホルダー・変数チェック

### 目的
動的に挿入される値のパターンが正しく維持されているか確認

### プラットフォーム別パターン

#### iOS
```
%@     文字列
%lld   整数（long long）
%d     整数
%.0f   浮動小数点（小数点なし）
```

**チェック例:**
```
英語: "Version %@"
日本語: "バージョン %@" ✓
日本語: "バージョン" ✗ (プレースホルダー欠落)
```

#### Android
```
%s        文字列
%d        整数
%1$s      位置指定文字列（1番目）
%2$d      位置指定整数（2番目）
```

**チェック例:**
```
英語: "%1$s selected %2$d items"
日本語: "%1$sが%2$d個選択しました" ✓
日本語: "%sが%d個選択しました" ✗ (位置指定が欠落)
```

#### Web (一般的なテンプレート)
```
{name}           単純な置換
{{variable}}     Mustache/Handlebars
${variable}      ES6テンプレート
$variable        一部のフレームワーク
```

#### ICU MessageFormat
```
{count, plural, one {1 item} other {# items}}
{gender, select, male {He} female {She} other {They}}
```

### チェックポイント

#### 4.1 プレースホルダーの数
原文と翻訳でプレースホルダーの数が一致しているか

```
英語: "Delete %d items?"
日本語: "%d個のアイテムを削除しますか？" ✓
日本語: "アイテムを削除しますか？" ✗ (プレースホルダー欠落)
```

#### 4.2 プレースホルダーの順序
位置指定がある場合、順序が適切か

```
英語: "%1$s deleted %2$d files"
韓国語: "%1$s님이 파일 %2$d개를 삭제했습니다" ✓
韓国語: "%2$d개 파일을 %1$s님이 삭제했습니다" ✓ (順序変更OK)
韓国語: "%s님이 파일 %d개를 삭제했습니다" ✗ (位置指定が欠落)
```

#### 4.3 数値と単位の間のスペース
言語慣習に合っているか

```
英語: "%.0f s" (スペースあり)
日本語: "%.0f秒" (スペースなし)
韓国語: "%.0f초" (スペースなし)
中国語: "%.0f秒" (スペースなし)
```

### チェック手順

1. プレースホルダーを含む文言を抽出
2. 原文と各翻訳でプレースホルダーの数を比較
3. 位置指定があれば、正しく維持されているか確認
4. 数値・単位の間のスペースが言語慣習に合っているか確認
5. 問題箇所を特定し、修正案を提示

## 5. 翻訳漏れチェック

### 目的
すべての文言が翻訳されているか確認

### チェック内容

#### 5.1 キーの存在確認
原文言語（通常は英語）に存在するキーがすべての対象言語に存在するか

**確認方法（JSON形式の場合）:**
```bash
# 英語のキー一覧を取得
jq '.strings | keys' en.json > en_keys.txt

# 日本語のキー一覧を取得
jq '.strings | keys' ja.json > ja_keys.txt

# 差分を確認
diff en_keys.txt ja_keys.txt
```

#### 5.2 空文字列チェック
翻訳が空文字列になっていないか

```json
// 問題例
{
  "welcome": {
    "en": "Welcome",
    "ja": ""  // ✗ 空文字列
  }
}
```

#### 5.3 TODOマーカーチェック
未翻訳マーカーが残っていないか

```bash
grep -r "TODO" **/*.json
grep -r "FIXME" **/*.json
grep -r "未翻訳" **/*.json
```

#### 5.4 コピー＆ペーストミス
英語のまま残っている箇所がないか

```json
// 問題例
{
  "cancel": {
    "en": "Cancel",
    "ja": "Cancel"  // ✗ 英語のまま
  }
}
```

**検出方法:**
```bash
# 日本語ファイル内で英単語が残っている箇所を検索
grep -E '[A-Za-z]{4,}' ja.json
```

### チェック手順

1. 対象言語を特定（例：en, ja, ko）
2. 各言語のキー一覧を抽出
3. 基準言語（en）と比較して欠落キーを特定
4. 空文字列、TODOマーカーを検索
5. 翻訳漏れリストを作成

## 6. 文字数・長さチェック

### 目的
UIに収まる適切な長さか確認

### 一般的な目安

| テキストタイプ | 推奨長 | 備考 |
|--------------|--------|------|
| ボタンラベル | 1-3語、10-20文字 | 極端に簡潔に |
| 画面タイトル | 2-5語、20-30文字 | 簡潔に |
| 短文メッセージ | 原文の80-150% | 言語による |
| 長文説明 | 原文の80-200% | 言語による |

### 言語別の傾向

**日本語・中国語:**
- 表意文字のため英語より短くなる傾向
- 漢字の圧縮効果

**韓国語:**
- 英語と同程度か少し長め

**ドイツ語・フィンランド語:**
- 英語より長くなる傾向
- 複合語が多い

### チェックポイント

#### 6.1 ボタンラベルの長さ
モバイルUIで収まるか

```
英語: "OK" (2文字)
日本語: "OK" (2文字) ✓
日本語: "よろしいですか？" (9文字) ✗ (長すぎる)

推奨: "了解" (2文字)
```

#### 6.2 画面タイトルの長さ
ナビゲーションバーに収まるか

```
英語: "Settings" (8文字)
日本語: "設定" (2文字) ✓
日本語: "アプリケーション設定画面" (12文字) ✗ (長すぎる)
```

#### 6.3 メッセージの長さ
ダイアログやトーストに収まるか

```
英語: "File saved successfully" (22文字)
日本語: "ファイルを保存しました" (11文字) ✓
日本語: "ファイルの保存処理が正常に完了いたしました" (22文字) △ (やや冗長)
```

### チェック手順

1. UIのタイプ別に文言を分類
2. 各文言の文字数をカウント
3. 推奨長を超える文言を特定
4. UIデザイナーと相談して許容範囲を確認
5. 必要に応じて短縮案を提案

## 7. 文化的配慮チェック

### 目的
各言語圏で不適切な表現がないか確認

### チェック内容

#### 7.1 文化・宗教的配慮
特定の文化や宗教に偏った表現がないか

**確認項目:**
- 宗教的シンボルや用語
- 文化固有の慣習への言及
- 政治的に敏感な表現

#### 7.2 市場適合性
各市場で受け入れられる表現か

**例（色の表現）:**
```
日本語: "ブラック", "ホワイト"
韓国語: "검정", "흰색"
中国語: "黑色", "白色"
```

#### 7.3 法的・規制的配慮
特にプライバシー関連の説明文

**確認項目:**
- GDPR準拠（EU市場）
- プライバシーポリシーの明確性
- データ収集の目的説明

**例（プライバシー許可）:**
```
英語: "We will use it to display photos in a slideshow."
日本語: "スライドショーに表示するために利用します"
→ 目的を明確に説明
```

#### 7.4 ジェンダーニュートラル表現
性別を特定しない表現になっているか

**例:**
```
避ける: "he/she"
推奨: "they", "the user"

避ける: "彼/彼女"
推奨: "ユーザー", "その人"
```

### チェック手順

1. 対象市場の文化的背景を調査
2. 法的要件（GDPR等）を確認
3. 文言を文化的観点からレビュー
4. 必要に応じて現地のネイティブスピーカーに確認
5. 問題箇所を特定し、代替案を提案

## 8. 自然さ・品質チェック

### 目的
機械翻訳的な不自然さがないか確認

### チェック内容

#### 8.1 文法的正確性
文法的に正しいか

**日本語の例:**
```
誤: "写真が選択されています" (助詞の誤り)
正: "写真が選択されています" ✓

誤: "ファイルを削除しますか" (句読点欠落)
正: "ファイルを削除しますか？" ✓
```

#### 8.2 自然な表現
ネイティブスピーカーにとって自然か

**機械翻訳的な例:**
```
機械的: "あなたは本当に削除したいですか？"
自然: "削除してもよろしいですか？"

機械的: "操作は成功しました"
自然: "完了しました"
```

#### 8.3 文脈適合性
文脈に適しているか

**例（エラーメッセージ）:**
```
不適切: "失敗" (冷たすぎる)
適切: "削除できませんでした" (丁寧で具体的)
```

#### 8.4 専門用語の正確性
専門用語が正しく使われているか

**例（技術用語）:**
```
誤: "認証を確認する" (誤用)
正: "認証する" または "本人確認する"

誤: "ダウンロードを実施"
正: "ダウンロードする"
```

### チェック手順

1. 機械翻訳を疑わせる不自然な表現を特定
2. ネイティブスピーカーの観点でレビュー
3. 文脈に即した自然な表現を提案
4. 可能であればネイティブスピーカーのレビューを受ける
5. 修正案をリストアップ

```markdown
- 対象言語: en, ja, ko
- 検出された問題: Y件（重大: A件、軽微: B件）

- [ファイル名:キー] "Cancel"が「キャンセル」「取消」で不統一
  - 推奨: すべて「キャンセル」に統一

### 2. 用語統一の問題（重大: X件）
- [ファイル名:キー] "Face ID"が"フェイスID"と表記
  - 推奨: "Face ID"（原文のまま）

### 3. 文体の問題（軽微: X件）
- [ファイル名:キー] ボタンラベルが「削除する」（動詞形）
  - 推奨: 「削除」（名詞形）に統一

### 4. プレースホルダーの問題（重大: X件）
- [ファイル名:キー] "%d"が翻訳で欠落
  - 推奨: "%d個のアイテム"

### 5. 翻訳漏れ（重大: X件）
- [ファイル名:キー] 日本語翻訳が存在しない
  - 推奨: 翻訳を追加

### 6. 文字数の問題（軽微: X件）
- [ファイル名:キー] ボタンラベルが25文字で長すぎる
  - 推奨: 10文字以内に短縮

### 7. 文化的配慮の問題（重大: X件）
- 該当なし

### 8. 自然さの問題（軽微: X件）
- [ファイル名:キー] "あなたは削除したいですか"は不自然
  - 推奨: "削除してもよろしいですか？"

### 良い点
- プライバシー関連の説明文が丁寧で適切
- 技術用語の統一が取れている

### 推奨事項
1. 「キャンセル」「取消」を統一する
2. プレースホルダー欠落を修正する
3. ボタンラベルを名詞形に統一する

### 参考情報
- 使用した翻訳ガイドライン: TRANSLATION_GUIDELINES.md
- 対象プラットフォーム: iOS
```
