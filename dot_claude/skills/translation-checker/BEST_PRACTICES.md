## 翻訳の一般原則

### 1. 一貫性を最優先

**原則:**
同じ概念には同じ用語を使用

**例:**
```
✓ 統一されている:
  File1: "Cancel" → "キャンセル"
  File2: "Cancel" → "キャンセル"
  File3: "Cancel" → "キャンセル"

✗ 不統一:
  File1: "Cancel" → "キャンセル"
  File2: "Cancel" → "取消"
  File3: "Cancel" → "中止"
```

**実践方法:**
- 翻訳開始前に用語集を作成
- 既存の翻訳パターンを調査
- 翻訳メモリツールを使用

### 2. 文脈を考慮

**原則:**
UIのどこで使われるかを理解して翻訳

**例:**
```
"Settings"の翻訳:

✓ 画面タイトル: "設定"
✓ ボタンラベル: "設定"
✓ メニュー項目: "設定"

文脈によって適切な長さや表現が変わる場合も:
✓ ナビゲーション: "設定"（簡潔）
✓ 説明文: "アプリケーション設定"（詳細）
```

**実践方法:**
- スクリーンショットで確認
- 実機/シミュレータで表示を確認
- デザインモックを参照

### 3. 簡潔さと明確さ

**原則:**
特にモバイルUIでは簡潔に、しかし明確に

**例:**
```
✓ 簡潔で明確:
  "削除しますか？" (9文字)

✗ 冗長:
  "本当にこのファイルを削除してもよろしいですか？" (23文字)

✗ 簡潔すぎて不明確:
  "削除？" (3文字)
```

**バランス:**
- ボタンラベル: 最も簡潔に
- 確認ダイアログ: 明確さを優先
- エラーメッセージ: 丁寧さと情報量のバランス

### 4. 文化的配慮

**原則:**
各言語圏の文化や慣習を尊重

**例:**
```
色の表現:
  英語: "Black", "White"
  日本語: "ブラック", "ホワイト"（カタカナが一般的）
  韓国語: "검정", "흰색"（韓国語固有語）
  中国語: "黑色", "白色"（漢字）

日付形式:
  英語: MM/DD/YYYY
  日本語: YYYY/MM/DD
  韓国語: YYYY. MM. DD.
```

**注意点:**
- 宗教的シンボルを避ける
- 政治的に敏感な表現を避ける
- ジェンダーニュートラルな表現を使用

### 5. ネイティブチェック

**原則:**
可能であればネイティブスピーカーのレビューを受ける

**レビューポイント:**
- 自然な表現か
- 文法的に正しいか
- 文化的に適切か
- UIに収まる長さか

**代替手段:**
- 複数の翻訳ツールで確認
- ネイティブスピーカーのコミュニティでフィードバック
- 既存の公式翻訳（Apple, Google等）を参考

## UI文言の原則

### 1. ユーザー中心

**原則:**
ユーザーが理解しやすい言葉を選ぶ

**例:**
```
✓ ユーザー視点:
  "写真を保存しますか？"
  "削除してもよろしいですか？"

✗ システム視点:
  "保存処理を実行しますか？"
  "データベースから削除しますか？"
```

**実践:**
- 専門用語を避ける
- ユーザーの行動に焦点を当てる
- "あなたの〜"を使う（適度に）

### 2. アクション指向

**原則:**
ボタンは何が起こるかを明確に

**例:**
```
✓ アクションが明確:
  "削除"（何を削除するか文脈から明らか）
  "写真を選択"（何を選択するか明示）

✗ 曖昧:
  "OK"（何が起こるか不明確）
  "はい"（何に「はい」なのか不明確）

改善:
  "削除" → "写真を削除"（文脈が不明確な場合）
  "OK" → "保存"（具体的なアクション）
```

### 3. 肯定的表現

**原則:**
否定形より肯定形を優先

**例:**
```
✓ 肯定的:
  "写真を保持"
  "変更を保存"

✗ 否定的:
  "削除しない"
  "保存しないで閉じる"

例外: 警告やエラー
✓ "削除できません"（事実を伝える）
```

### 4. エラーは丁寧に

**原則:**
エラーメッセージは特に丁寧に

**例:**
```
✓ 丁寧で情報的:
  "ネットワークに接続できませんでした。接続を確認してください。"
  "お手数ですが、もう一度お試しください。"

✗ 冷たい:
  "エラー"
  "失敗"
  "ネットワークなし"

✗ 過度に技術的:
  "ERR_CONNECTION_REFUSED"
  "HTTP 404 Not Found"
```

**構成:**
1. 何が起こったか（事実）
2. なぜ起こったか（可能であれば）
3. どうすればよいか（対処法）

### 5. 一貫したトーン

**原則:**
アプリ全体で統一されたトーン

**例:**
```
✓ 一貫している:
  ボタン: "削除", "保存", "キャンセル"（すべて名詞形）
  メッセージ: "保存しました", "削除しました"（すべてです・ます調）

✗ 不統一:
  ボタン: "削除", "保存する", "キャンセル"（名詞形と動詞形が混在）
  メッセージ: "保存した", "削除しました"（敬語レベルが不統一）
```

## 避けるべきパターン

### 1. 機械翻訳の直訳

**問題:**
不自然で理解しにくい

**例:**
```
英語: "You have 5 new messages"

✗ 機械翻訳:
  日本語: "あなたは5つの新しいメッセージを持っています"
  韓国語: "당신은 5개의 새 메시지를 가지고 있습니다"

✓ 自然な翻訳:
  日本語: "新しいメッセージが5件あります"
  韓国語: "새 메시지 5개가 있습니다"
```

### 2. 過度な敬語

**問題:**
冗長でUIに収まらない

**例:**
```
✗ 過度:
  "大変恐れ入りますが、削除してもよろしいでしょうか？"（27文字）

✓ 適切:
  "削除してもよろしいですか？"（13文字）
```

### 3. 文化的に不適切な表現

**問題:**
特定の地域で誤解や不快感

**例:**
```
✗ 問題あり:
  宗教的シンボルの使用
  政治的に敏感な表現
  性別を特定する表現（"he/she"）

✓ 適切:
  ニュートラルなシンボル
  政治的に中立な表現
  ジェンダーニュートラルな表現（"they", "the user"）
```

### 4. 一貫性のない用語

**問題:**
ユーザーの混乱を招く

**例:**
```
✗ 不統一:
  画面1: "ログイン"
  画面2: "サインイン"
  画面3: "認証"

✓ 統一:
  すべて: "ログイン"
```

### 5. プレースホルダーの誤用

**問題:**
表示が壊れる

**例:**
```
英語: "Delete %d items?"

✗ プレースホルダー欠落:
  日本語: "アイテムを削除しますか？"（数が表示されない）

✓ 正しい:
  日本語: "%d個のアイテムを削除しますか？"
```

## ツールとワークフロー

### 推奨ツール

#### 翻訳管理
- **Lokalise**: クラウドベースの翻訳管理
- **Crowdin**: コラボレーション翻訳プラットフォーム
- **Phrase**: エンタープライズ翻訳管理

#### 品質チェック
- **SDL Trados**: 翻訳メモリとQAツール
- **Xbench**: 翻訳品質チェック
- **ApSIC Xbench**: 用語統一チェック

#### プレビュー
- **実機/シミュレータ**: 実際の表示確認
- **BrowserStack**: Web翻訳の確認
- **Figma/Sketch**: デザインモックでのプレビュー

### ワークフロー例

#### 1. 準備フェーズ
```
1. 要件確認
   - ターゲット言語
   - 納期
   - 品質基準

2. 環境準備
   - 翻訳ツールのセットアップ
   - 用語集の準備
   - スタイルガイドの確認

3. 既存翻訳の調査
   - パターン分析
   - 用語抽出
```

#### 2. 翻訳フェーズ
```
1. 初回翻訳
   - 用語集に従う
   - プレースホルダーを維持
   - 文脈を考慮

2. セルフレビュー
   - 一貫性確認
   - 誤字脱字チェック
   - プレースホルダーチェック

3. クロスチェック
   - 他の翻訳者とレビュー
   - 用語統一確認
```

#### 3. レビューフェーズ
```
1. 自動チェック
   - プレースホルダー検証
   - 文字数チェック
   - 翻訳漏れ検出

2. 手動レビュー
   - 文体確認
   - 自然さチェック
   - 文化的配慮確認

3. ネイティブレビュー
   - 最終品質確認
```

#### 4. 実装・確認フェーズ
```
1. 実装
   - 翻訳ファイルをプロジェクトに統合
   - ビルド確認

2. 実機確認
   - UIフィット確認
   - 動作確認

3. フィードバック
   - 問題点の修正
   - 改善提案
```

### 継続的改善

#### フィードバックループ
```
ユーザーフィードバック
  ↓
問題点の特定
  ↓
用語集・スタイルガイド更新
  ↓
翻訳の改善
  ↓
品質向上
```

#### メトリクス
- 翻訳漏れ率
- 用語統一率
- ネイティブレビュー合格率
- ユーザーフィードバックスコア

## 言語別のベストプラクティス

### 日本語
- 漢字を適度に使用（読みやすさとのバランス）
- 長音符を統一（データー vs データ）
- 句読点を適切に使用

### 韓国語
- 띄어쓰기を正確に
- 존댓말を統一
- 한자어와 고유어の適切な使い分け

### 英語
- Title Caseは避ける（Sentence caseを推奨）
- Oxfordコンマは統一
- アメリカ英語/イギリス英語を統一

### 中国語
- 简体字/繁体字を統一
- 标点符号を正確に使用（全角/半角）
- 地域による用語の違いに注意

## まとめ

**翻訳品質の3要素:**
1. **正確性**: 意味が正しく伝わる
2. **一貫性**: 用語とスタイルが統一
3. **自然さ**: ネイティブスピーカーにとって自然

**成功の鍵:**
- 準備を怠らない
- 文脈を理解する
- 一貫性を保つ
- ネイティブの視点を得る
- 継続的に改善する
