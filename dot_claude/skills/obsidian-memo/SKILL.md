# Obsidian Memo

セッションで得た汎用的な技術知識をObsidianノートとして記録する。

## 対象ディレクトリ

`~/obsidian-notes/memo`

## ワークフロー

### 1. 知識の抽出

セッションの会話履歴から、以下の基準で汎用的な知識を抽出する：

**含めるべき内容:**
- 技術的なベストプラクティス
- 問題解決パターンと解決策
- ツールやライブラリの使い方
- 設計パターンやアーキテクチャの考え方
- デバッグテクニック
- パフォーマンス最適化の手法

**除外すべき内容（必須）:**
- プロジェクト名、会社名、個人名
- ファイルパス、URL、IPアドレス
- APIキー、認証情報、トークン
- 内部仕様や非公開の実装詳細
- 特定のコードベースに依存する情報

### 2. トピックの分離

抽出した知識をトピックごとに分離する。1つのノートには1つの明確なトピックのみを含める。

トピック例：
- 「Swiftの並行処理パターン」
- 「Git rebaseのベストプラクティス」
- 「iOSのメモリ管理」

### 3. 既存ファイルの確認

ノートを作成する前に、類似のファイルが存在するか確認する：

```bash
ls ~/obsidian-notes/memo/
```

類似ファイルが存在する場合は内容を読み、追記が適切か判断する。

### 4. ファイル作成または追記

**新規作成の場合:**

ファイル名フォーマット: `YYYYMMDD-Title.md`
- 日付は今日の日付
- Titleは日本語または英語（ケバブケース、スペース不可）

**追記の場合:**

既存ファイルの構造を維持しつつ、新しい知見を適切なセクションに追加する。

### 5. ノートのフォーマット

```markdown
# タイトル

## 概要

[2-3文で要約]

## 詳細

### サブトピック1

[説明と具体例]

### サブトピック2

[説明と具体例]

## 参考

- [外部リンクや参考資料があれば]
```

## 記述スタイル

プロのテクニカルライターとして以下を遵守：

- 明確で簡潔な文章
- 具体的なコード例を含める（汎用化したもの）
- 専門用語は必要に応じて説明を添える
- 箇条書きを活用して読みやすくする
- 主観的な表現を避け、事実ベースで記述

## 完了確認

ノート作成後、以下を確認：

1. 機密情報が含まれていないこと
2. プロジェクト固有の情報が含まれていないこと
3. 他のプロジェクトでも再利用可能な汎用的な内容であること
4. ファイル名がフォーマットに従っていること
